### １．ドメイン駆動設計（Domain-Driven Design, DDD）
DDDは、2003年にエリック・エヴァンス氏が提唱した**「複雑なビジネスソフトウェアの設計手法」です。単にコードをきれいに書く技術ではなく、「ビジネスのルール（ドメイン）を正しく理解し、それをそのままソフトウェアの形にする」**ことを目指します。

---

### ２．書籍要約（エリック・エヴァンス、ドメイン駆動設計）
#### 第1部：ドメインモデルを機能させる
ここでは「そもそもなぜモデルが必要なのか？」という哲学と、開発者とビジネス側のコミュニケーションについて説いています。

- 中心テーマ: ユビキタス言語
- エヴァンスの主張: モデルはホワイトボードの図ではなく、共通言語として会話の中に存在し、そのままコードに投影されなければならない。

#### 第2部：モデル駆動設計の構成要素
エンジニアに最も馴染み深い「戦術的設計（ビルディングブロック）」の章です。

- 主なパターン:
    - エンティティ / 値オブジェクト: 物をどう識別するか。
    - サービス: 状態を持たない操作。
    - モジュール: 関連する概念をどう包むか。

- エヴァンスの主張: データベースの都合（テーブル構造）ではなく、ドメインの概念（業務ルール）をベースにオブジェクトを設計せよ。

#### 第3部：より深い洞察へ向かうリファクタリング
ここからがエヴァンス本の「真骨頂」です。単なる整理整頓ではないリファクタリングを論じます。

- 中心テーマ: 深いモデル / しなやかな設計
- エヴァンスの主張: * 開発の途中で「あ、この概念はもっとシンプルに表現できる！」という気づき（ブレイクスルー）が来る。
    - その時、コードをドメインの核心に合わせて大胆に書き直す（リファクタリング）ことで、設計が「しなやか」になり、変更に強くなる。

#### 第4部：戦略的設計
最後のパートは、巨大なシステムをどう統治するかという、アーキテクト視点の話です。

- 主な概念:
    - 境界づけられたコンテキスト: 意味が混ざらないようにモデルを分離する。
    - コンテキストマップ: 分離されたモデル同士の関係性（協力、依存など）を図示する。
    - 蒸留（Distillation）: 最も重要な「コア・ドメイン」を見極め、そこにリソースを集中させる。
- エヴァンスの主張: すべてを完璧に作ろうとするな。ビジネスの成功に直結する「核心（コア）」を特定し、そこを徹底的に磨き上げろ。

|パート|視点|目的|
|---|---|---|
|第1部|コミュニケーション|開発者とビジネス側の言葉を一致させる
|第2部|実装パターン|業務ルールを正しくコードに落とし込む
|第3部|設計の質|複雑さを取り除き、柔軟なコードにする
|第4部|全体戦略|巨大なシステムを分割し、核心に集中する

### ３．DDD（ドメイン駆動設計）新旧比較：エヴァンス原典 vs 現代

| 比較項目 | エヴァンスのDDD (2003年/原典) | 現代のDDD (2020年代〜) |
| :--- | :--- | :--- |
| **中心的な関心** | **静的な構造（状態）**：オブジェクトが「どうあるか」 | **動的な事象（イベント）**：ビジネスで「何が起きたか」 |
| **データの整合性** | **強い整合性**：1つのトランザクションで全て完了させる | **結果整合性**：イベント連鎖により、最終的に整えばOK<br/>イベント中にエラーが発生した場合、完了したトランザクションのデータの一貫性がなくなる。この場合は補償トランザクション（打ち消すための逆の処理。ビジネス的な解決で対応する。） |
| **データの管理方法** | **単一モデル**：読み書きを同じモデルで行う | **CQRS**：書き込み（Command）と読み取り（Query）を分離 |
| **主な構成要素** | エンティティ、集約、リポジトリ、サービス | **ドメインイベント、プロジェクション、ポリシー** |
| **アーキテクチャ** | レイヤードアーキテクチャ | **クリーン、オニオン、ヘキサゴナル** |
| **通信スタイル** | 同期処理（メソッド呼び出し中心） | **非同期処理（イベント駆動・メッセージング中心）** |
| **主な課題解決** | 1つの巨大なシステムの複雑さの解消 | **分散システム（マイクロサービス）間の連携** |

#### 補足：現代における用語の繋がり
現代のDDDでは、これらの用語は以下のように組み合わさって機能します。

1. **ドメインイベント**: 変化の起点（「住所が変更された」という事実）。
1. **プロジェクション**: イベントを受け取って、表示用のデータを作成する（投影）。
1. **結果整合性**: 投影には少し時間がかかるかもしれないが、ユーザーが次に画面を開く頃には整合している。

#### ３．１ 具体的な比較
##### （１）ユースケース
1. 会員が新規入会する
1. 会員が入館する
1. 会員がトレーニングする
1. 会員がシャワーを浴びる
1. 会員が退館する

#### （２）比較：スポーツジム会員管理における新旧DDD比較

| 比較項目 | エヴァンスのDDD (2003年/原典) | 現代のDDD (2020年代〜) |
| :--- | :--- | :--- |
| **中心的な関心** | **状態の正しさ**：今「入館中」か「シャワー中」か、整合したスナップショットを保持すること。 | **出来事の連鎖**：「入館した」「シャワーを浴びた」という事実を起点に、関連サービスを動かすこと。 |
| **データの整合性** | **強い整合性**：入館処理と同時に、利用履歴保存やロッカー確保を1つの処理（Transaction）で完結させる。 | **結果整合性**：入館は即完了。その数秒後に、別プロセスで滞在タイマーが動き出し、統計が更新される。 |
| **データの管理方法** | **単一の会員モデル**：1つのMemberテーブルに、入会情報も現在の居場所もすべて集約して管理する。 | **CQRS**：更新はMember集約で行い、館内状況の表示（照会）は専用の読み取り用モデルを参照する。 |
| **主な構成要素** | 会員（エンティティ）、入館（メソッド）、会員リポジトリ。 | 会員入館済み（イベント）、滞在監視（ポリシー）、利用状況投影（プロジェクション）。 |
| **アーキテクチャ** | **レイヤード**：UI→アプリ→ドメイン→インフラの順で、「手続き」の階層を整理する。 | **クリーン/イベント駆動**：ドメインを中心に、外部（センサー、メール、DB）をプラグインとして扱う。 |
| **通信スタイル** | **同期（命令）**：入会処理の中から「メール送信関数」を直接呼び出し、返り値を待つ。 | **非同期（反応）**：入会イベントを放流し、メール担当がそれを「検知」して勝手に動く。 |
| **主な課題解決** | 巨大な会員管理ロジックがスパゲティコード（密結合）になるのを防ぐ。 | ジム、アプリ、自動改札、清掃ロボットなど、多様なシステムを疎結合に連携させる。 |

### ４．現代のDDD（2020年代〜）重要用語まとめ

#### 1. 「出来事」と「状態」に関する用語
| 用語 | 読み | 意味・役割 | スポーツジムでの具体例 |
| :--- | :--- | :--- | :--- |
| **Domain Event** | ドメインイベント | ビジネス上、価値のある「過去の事実」。過去形で命名する。 | `MemberEntered`（会員が入館した） |
| **Event Sourcing** | イベントソーシング | 現在の状態ではなく、発生した全イベントの「履歴」を保存する手法。 | 10:00入館、11:00シャワー...という履歴から現在の場所を特定する。 |
| **Event Store** | イベントストア | ドメインイベントを永続化するための専用DB。 | 全ての「出来事」が消えずに記録されているログ基盤。 |

#### 2. 「協調」と「連鎖」に関する用語
| 用語 | 読み | 意味・役割 | スポーツジムでの具体例 |
| :--- | :--- | :--- | :--- |
| **Policy** | ポリシー | 「〜が起きたら、〜する」という反応のルール（接着剤）。 | 「退館したら、滞在時間を計算する」というビジネスルール。 |
| **Saga** | サーガ | 複数のサービスにまたがる長い業務プロセスと、エラー時の制御。 | 入会処理中にエラーが起きたら、予約したロッカーを自動キャンセルする。 |
| **Compensating Transaction** | 補償トランザクション | 失敗したときに、すでに行われた処理を「打ち消す」ための逆処理。 | 「ロッカー予約の取り消し」処理そのもの。 |
| **Choreography** | コレオグラフィ | 各サービスがイベントを自律的に検知して動く協調スタイル。 | 改札が動くと、アプリと清掃ロボットが勝手に反応する様子。 |

#### 3. 「データの最適化」に関する用語
| 用語 | 読み | 意味・役割 | スポーツジムでの具体例 |
| :--- | :--- | :--- | :--- |
| **CQRS** | シーキューアールエス | 書き込み用モデルと読み取り用モデルを完全に分離する設計。 | 「入館を記録するDB」と「混雑状況を表示するDB」を分ける。 |
| **Projection** | プロジェクション | イベント履歴を加工して、別のデータ形式（表示用など）に変換すること。 | 膨大な入退館イベントから「現在の館内人数：15人」という数値を導き出す。 |
| **Read Model** | リードモデル | 画面表示に最適化された、読み取り専用のデータ。 | AngularのSignalに流し込むための「本日の消費カロリー合計」データ。 |

#### 4. 「境界の繋ぎ」に関する用語
| 用語 | 読み | 意味・役割 | スポーツジムでの具体例 |
| :--- | :--- | :--- | :--- |
| **Anticorruption Layer (ACL)** | 腐敗防止層 | 外部の古い・汚いデータが、自ドメインを汚さないための変換層。 | 外部の古いクレカ決済APIの難解な応答を、綺麗なデータに変換する。 |
| **Context Map** | コンテキストマップ | 各ドメイン（コンテキスト）がどう繋がっているかを示した全体図。 | ジム、アプリ、改札、ロボットの連携ルートを記した地図。 |
| **Upstream / Downstream** | 上流 / 下流 | データの供給側（上流）と、その影響を受ける側（下流）の関係。 | 会員管理（上流）でプランが廃止されると、アプリ（下流）も影響を受ける。 |


### ５.パッケージ構成
システムの第一階層は技術用語ではなく、ビジネスの境界（コンテキスト）で分割します。

#### （１）現代のDDDとAngularの設計思想を組み合わせた、理想的なフォルダ構成のコードブロックです。

```text
src/app/
  ├── membership/          # 会員管理コンテキスト（入会、契約、支払）
  ├── access-control/      # 入退館コンテキスト（自動改札、ゲート制御、滞在管理）
  ├── facility-cleaning/   # 施設清掃コンテキスト（シャワー室状況、清掃ロボット）
  └── shared/              # 共通基盤（各コンテキストで使う共通ユーティリティ）
```

#### （２）コンテキスト内部の構成（ここで現代DDD用語が登場する）各コンテキスト（例：access-control）の内部で、役割に応じた現代DDDの用語を配置します。
```text

  ├── membership/          # 会員管理コンテキスト（入会、契約、支払）
  │    ├── projections/    # Projections（イベントを検知してSignalを更新）
  │    ├── read-models/    # Read Models（Signalで扱う表示専用のデータ型）
  │    └── components/     # Angular Components（画面、テンプレート）
  ├── access-control/      # 入退館コンテキスト（自動改札、ゲート制御、滞在管理）

  ├── membership//
```
